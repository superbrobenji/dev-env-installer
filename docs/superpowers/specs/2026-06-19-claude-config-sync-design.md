# Claude Code Config Sync Design

**Date:** 2026-06-19
**Status:** Approved
**Author:** Benjamin Swanepoel

## 1. Purpose

Make Claude Code configuration (skills, plugins, model preference) reproducible across machines via the existing dev-env-installer. Running `install.sh` on a fresh machine should produce the same Claude Code environment — same skills available, same plugins installed, same model — as on the primary machine.

## 2. Goals

- Skills tracked in the dotfiles repo alongside other config files.
- Plugins installed idempotently via the `claude` CLI.
- `settings.json` written/merged on each install, preserving machine-specific keys.
- Fits existing installer pattern (`check` + `install` functions, added to `TOOL_ORDER`).
- ClearScore-specific skills excluded from the personal dotfiles repo entirely.

## 3. Non-Goals

- Syncing `~/.claude/sessions/`, `history.jsonl`, or any runtime state.
- Syncing the plugin cache (downloaded data, always re-fetched).
- Tracking `settings.json` in git (it contains machine-specific absolute paths).
- A separate `claude-config` repo (overkill for a handful of markdown files).

## 4. What Gets Synced

### 4.1 Skills (dotfiles repo)

Added to `~/.dotfiles` and copied to `~/.claude/skills/` by `checkout_dotfiles`:

```
.claude/skills/find-skills/SKILL.md
.claude/skills/slack-messaging/SKILL.md
.claude/skills/task-parallelism/SKILL.md
.claude/skills/technical-design-documentation/SKILL.md
.claude/skills/technical-design-documentation/references/*
```

`checkout_dotfiles` already handles arbitrarily nested paths via `git ls-tree -r HEAD --name-only` + `mkdir -p` + `cp -a`. No changes to `dotfiles.sh` required.

Excluded (ClearScore-specific, not tracked):
- `surface-engineer`
- `surface-manual-tester`

`slack-messaging/SKILL.md` description contains "at ClearScore" — strip that phrase before committing to the dotfiles repo.

### 4.2 Settings (generated by `claude.sh`)

`~/.claude/settings.json` is NOT tracked in the dotfiles repo. It contains machine-specific absolute paths (the `statusLine` command embeds `$USER` and a plugin cache commit SHA that changes per install). `claude.sh` owns the merge logic.

Portable keys written/updated by the installer:

```json
{
  "model": "claude-opus-4-7",
  "enabledPlugins": {
    "typescript-lsp@claude-plugins-official": true,
    "superpowers@claude-plugins-official": true,
    "superpowers@superpowers-marketplace": true,
    "caveman@caveman": true,
    "lua-lsp@claude-plugins-official": true,
    "pyright-lsp@claude-plugins-official": true
  },
  "extraKnownMarketplaces": {
    "superpowers-marketplace": {
      "source": { "source": "github", "repo": "obra/superpowers-marketplace" }
    },
    "caveman": {
      "source": { "source": "github", "repo": "JuliusBrussee/caveman" }
    }
  }
}
```

Existing keys not listed above (e.g. `statusLine`) are preserved.

Merge is implemented as a Python inline heredoc — Python is guaranteed present by the time `claude_install` runs.

### 4.3 Plugins (installed by `claude.sh`)

Marketplaces added first (idempotent — skip if already in `claude plugin marketplace list`):
- `obra/superpowers-marketplace` → name `superpowers-marketplace`
- `JuliusBrussee/caveman` → name `caveman`

Plugins installed (idempotent — skip if already in `claude plugin list`):
- `typescript-lsp@claude-plugins-official`
- `superpowers@claude-plugins-official`
- `superpowers@superpowers-marketplace`
- `caveman@caveman`
- `lua-lsp@claude-plugins-official`
- `pyright-lsp@claude-plugins-official`

## 5. File Changes

### 5.1 New: `installers/claude.sh`

```
claude_check()          → command -v claude >/dev/null 2>&1
claude_install()        →
  1. npm install -g @anthropic-ai/claude-code
  2. _claude_merge_settings   (Python inline heredoc)
  3. _claude_add_marketplaces (skips already-present)
  4. _claude_install_plugins  (skips already-installed)
```

### 5.2 Modified: `install.sh`

Add `claude` to `TOOL_ORDER` after `node` (npm dependency):

```bash
TOOL_ORDER=(
  ...
  node
  claude     # new
  tree_sitter_cli
  ...
)
```

`claude` is not added to `FATAL_TOOLS` — a failed plugin install should not abort the full run.

### 5.3 No changes: `dotfiles.sh`

`checkout_dotfiles` handles the new `.claude/skills/` paths automatically.

### 5.4 External: `~/.dotfiles` repo

The four skill directories above are committed and pushed to `github.com/superbrobenji/dotfiles`. This is a separate action from the installer changes.

## 6. Idempotency

| Step | Already-done guard |
|------|--------------------|
| `claude_check` | `command -v claude` — whole installer skipped |
| `_claude_merge_settings` | Python merge preserves existing keys; safe to re-run |
| `_claude_add_marketplaces` | Check `claude plugin marketplace list` output before adding |
| `_claude_install_plugins` | Check `claude plugin list` output before installing each |
| Skills copy | `checkout_dotfiles` backs up before overwrite; idempotent |

Re-running with `--only claude` re-runs all sub-steps (useful after adding a new plugin).

## 7. Error Handling

- `claude_install` returns non-zero if `npm install` fails; logged as a failed tool.
- Marketplace/plugin failures are warned but do not abort (`|| warn "..."` pattern).
- Settings merge failure is fatal within `claude_install` (misconfigured Claude is worse than no Claude config).
